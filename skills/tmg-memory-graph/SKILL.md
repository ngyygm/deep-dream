# TMG Memory Graph
TMG 是 Agent 的长期记忆，不是可选日志。
默认原则只有一句话：**凡是你读过、说过、做过、想过，并且未来可能影响行动或理解的内容，都应该进入记忆。**

## 使用场景
当任务涉及以下任一目标时，使用本技能：
- 让 Agent 记住内容。
- 让 Agent 回忆历史上下文。
- 在继续旧任务前恢复记忆。
- 在阅读文档后把全文和结论存入记忆。
- 在会话结束、阶段结束、心跳同步时批量写入记忆。
- 启动、检查、调用 TMG 服务。

## 这份技能要解决什么
看完本文件后，你应当立刻知道三件事：
1. 什么内容必须写入记忆。
2. 在什么时机必须调用 `find` 或 `remember`。
3. 如何用最小成本把记忆写对，而不是零碎乱记。

## 记什么

把下面四类内容视为默认记忆对象：

| 类别 | 必须记录的内容 |
|---|---|
| 已读 | 文档全文、论文、网页、需求、配置、报错、日志、代码片段、运行结果 |
| 已说 | 用户说了什么，你回复了什么，双方达成了什么约定 |
| 已做 | 查了什么、改了什么、运行了什么、部署了什么、决定了什么 |
| 已想 | 计划、假设、疑虑、判断依据、方案比较结论、下一步打算 |

如果不确定该不该记，默认答案是：**记。**

## 什么时候调用

### 必须先 `find` 的时机

在下面场景，先检索记忆再继续工作：

- 新会话开始时。
- 继续一个中断过的任务时。
- 用户问“之前做过什么/说过什么/决定了什么”时。
- 开始读取新文档前，想知道它和既有上下文有什么关系时。
- 准备做重大决策前，需要先回忆历史约束、偏好、失败记录时。

推荐查询：

- `我最近在做什么`
- `之前关于 <主题> 讨论过什么`
- `这个项目上次停在哪里`
- `与 <文件/任务/用户目标> 相关的历史决策`

### 必须 `remember` 的时机

在下面场景，必须写入记忆：

1. 读完一份文档后。
2. 完成一段有意义的对话后。
3. 完成一个动作后。
4. 形成重要想法或决策后。
5. 会话结束、任务切换、心跳同步时。

把这五类展开如下：

| 时机 | 写入内容 |
|---|---|
| 读完文档 | 文档全文 + 你为什么读它 + 读后结论 |
| 完成对话 | 用户目标、你的回答、结论、未决问题 |
| 完成动作 | 改了什么、执行了什么、结果如何、失败原因 |
| 形成想法 | 新假设、风险判断、方案取舍、理由 |
| 结束阶段 | 本阶段完整时间线，不要只写摘要 |

## 什么时候不要立刻写

不要每句话、每个小操作都调用一次 `remember`。

正确做法：

- 在工作中持续积累事件。
- 攒成一个有意义的批次再写入。
- 一次写入一段按时间排序的叙事文本。

错误做法：

- 逐句写。
- 只写一句“我刚刚看了一个文件”。
- 文档只存摘要不存原文。

## 写入纪律

### 1. 写真实时间

`event_time` 记录的是事件实际发生时间，不是你发送请求的时间。

### 2. 写完整上下文

只写“做了 X”不够，至少补齐：

- 为何做。
- 怎么做。
- 结果怎样。
- 接下来怎么办。

### 3. 文档默认存全文

只要你读了文档，并且它可能影响后续理解或决策，就应把全文写入记忆。

### 4. 思考也要存

这里的“想过”不是空泛自言自语，而是对未来有价值的思考痕迹，例如：

- `我倾向于方案 A，因为 B 的维护成本更高`
- `我怀疑错误来自配置，而不是代码逻辑`
- `我下一步要先验证接口，再改实现`

### 5. 记忆是叙事，不是标签堆

TMG 处理的是自然语言时间叙事。不要发明复杂分类体系，不要手工维护多套标签树。

## 标准工作流

### 会话开始

1. 检查服务是否可用：`GET /health`
2. 如果服务未启动，先启动 TMG。
3. 调用 `POST /api/find`，查询与当前任务相关的历史上下文。

### 会话进行中

1. 持续收集：已读、已说、已做、已想。
2. 先工作，不要被频繁写记忆打断。
3. 遇到一个自然阶段结束时，批量写入一次。

### 会话结束或任务切换

1. 把这一阶段发生的事情整理成时间叙事。
2. 调用 `GET /api/remember`（查询参数传入正文）一次性写入。
3. 若有异步任务，记下 `task_id`，稍后检查状态。

## 阅读文档时的特殊规则

阅读是最容易遗漏记忆的环节，所以单独强调：

1. 开始阅读前，可先 `find` 相关背景。
2. 阅读过程中，保留文档完整文本。
3. 阅读结束后，必须 `remember`。
4. 写入内容至少包括：
   - 文档标题或主题
   - 你阅读它的原因
   - 文档全文
   - 你的结论、疑问、后续动作

如果只是存一句“我读了一篇论文”，这不算合格记忆。

## 对话时的特殊规则

对话不仅要记“用户说了什么”，还要记：

- 用户真正想达成什么。
- 你承诺了什么。
- 哪些约束被明确了。
- 哪些结论已经达成。
- 哪些问题仍未解决。

## 动作时的特殊规则

凡是实际执行过的行为，都应在阶段结束后进入记忆，例如：

- 读了哪些文件。
- 改了哪些文件。
- 跑了哪些命令。
- 看到了什么错误。
- 采用了什么修复策略。
- 最终成功还是失败。

## 决策与思考时的特殊规则

只记录结果不够，决策理由也要存。

优先记录这些高价值思考：

- 方案取舍。
- 风险判断。
- 对问题成因的假设。
- 对下一步行动的计划。
- 为什么放弃某条路线。

## 最小 API 集

你只需要记住四个接口：

| 用途 | 接口 |
|---|---|
| 健康检查 | `GET /health` |
| 写入记忆 | `GET /api/remember`（`text` 或 `text_b64`） |
| 查询记忆 | `POST /api/find` |
| 查询写入状态 | `GET /api/remember/status/<task_id>` |

默认地址通常是 `http://127.0.0.1:16200`。

**测试库与磁盘布局**：联调时建议 `storage_path` 使用仓库内 **`graph/tmg_storage`**（格式最新、含 `originals/` 等）。每次 remember 会在 `originals/` 落一份原文 txt，响应里的 `original_path` 与 `remember_journal/{task_id}.json` 与之对应；**检索图内容以 HTTP Find/原子接口为主**，不要仅依赖扫盘读 txt。更细的目录说明、**按目的选接口的表**、以及 **Agent 用代码多步组合查询** 的说明见 `reference.md`。

## Remember 请求规则

**仅使用 GET**，查询参数至少包含以下之一：

| 参数 | 说明 |
|---|---|
| `text` | 与 `text_b64` 二选一，自然语言叙事（URL 编码） |
| `text_b64` | 与 `text` 二选一，UTF-8 正文的标准 Base64（长文本推荐） |
| `event_time` | 强烈建议传，ISO 8601，事件实际发生时间 |
| `source_name` / `doc_name` | 可选，来源标识 |
| `load_cache_memory` | 可选，`true`/`false` |

关键规则：

- **不要再用 POST / JSON body** 调用 remember。
- 不要传 `file_path`；不要用 multipart。
- 默认异步：立即返回 `task_id`（HTTP 202），后台入队处理；服务默认 **多线程**，处理 remember 期间 **仍可响应 find**。
- 未完成任务会写入 `storage_path/remember_journal/`；进程崩溃或重启后，**queued / running** 会从 `originals/` 原文 **自动重新入队**继续处理（整段流水线重跑）。
- 需要时再轮询 `GET /api/remember/status/<task_id>`。

## Find 请求规则

推荐默认带：

```json
{
  "query": "我最近在做什么",
  "expand": true,
  "max_entities": 10,
  "max_relations": 20
}
```

`find` 负责召回，最终筛选和判断由你完成。

## 服务启动速记

项目根目录：`Temporal_Memory_Graph/`

常用步骤：

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp service_config.example.json service_config.json
python service_api.py --config service_config.json
```

启动前检查：

- `service_config.json` 已补全。
- `storage_path` 可写。
- `llm.think` 建议设为 `false`。
- 服务没在重复启动。

详细接口说明见 `reference.md`。

## 写入格式

每次 `remember` 的 `text` 应是一段按时间排列的忠实叙事。

格式模板见 `templates/remember-format.md`。

按需读取示例：

- 工作过程：`examples/work-session.md`
- 读完文档后写入：`examples/read-document.md`
- 当日复盘：`examples/daily-reflection.md`
- 阅读前预存：`examples/before-reading.md`

## 禁止事项

- 不要逐句调用 `remember`。
- 不要只存摘要而丢掉关键原文。
- 不要在知道真实时间时省略 `event_time`。
- 不要把 `find` 的结果原样当结论。
- 不要把记忆系统当“可选附加功能”。

## 一句话准则

**开始工作前先回忆，工作过程中积累，阶段结束后批量写入；把读过的、说过的、做过的、想过的，都按时间变成记忆。**
