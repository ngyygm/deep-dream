# 基于LLM裁判引导的树状搜索Prompt自动优化方法研究

## 一、研究题目

**面向多步骤NLP Pipeline的LLM裁判引导树状搜索Prompt自动优化框架**

### 英文标题

*A Tree-Search Prompt Optimization Framework Guided by LLM-as-Judge for Multi-Step NLP Pipelines*

---

## 二、研究背景与内容

### 2.1 问题背景

大语言模型（LLM）在信息抽取、知识图谱构建等结构化任务中的表现高度依赖于Prompt的设计质量。然而，当前Prompt工程实践存在三个核心瓶颈：

**（1）手工试错成本高昂。** 生产级NLP Pipeline通常包含多个串联步骤（如：文本摘要 → 实体抽取 → 关系抽取 → 实体对齐），每个步骤的Prompt需要独立调优，且步骤间存在耦合效应——上游Prompt的输出质量直接影响下游的输入分布。人工逐步骤调试耗时且不可复现。

**（2）A/B测试范式探索效率低。** 现有Prompt优化方法（包括DSPy等框架）本质上采用**平面搜索**——预定义若干候选变体，逐一评估，择优录取。这种范式有两个结构性缺陷：
- 搜索方向是**预先设定**的，无法根据中间结果动态调整
- 当变体效果不理想时，只能重新人工设计新变体，缺乏**自动回溯与分支**机制

**（3）弱模型无法自评。** 部署端通常使用轻量模型（如GLM-4-Flash）以控制推理成本，这类模型的输出质量评估无法依赖其自身——它既无法判断自己的抽取是否完整，也无法识别格式错误。需要一个**更强的外部裁判**。

### 2.2 研究内容

本研究提出**LLM裁判引导的树状搜索Prompt自动优化框架**（Tree-Search Prompt Optimization with LLM-as-Judge，简称**TreePrompt**），核心思想是：

> 将Prompt优化建模为**树状搜索问题**——每个节点代表一个Prompt变体，节点评估由强模型（裁判）执行，根据评估结果动态生成子节点（新变体），支持分支探索与失败回溯，直到找到满足质量标准的收敛节点。

具体研究内容包含以下四个方面：

#### （1）LLM裁判评估体系

设计面向多步骤NLP Pipeline的**步骤专属评估标准**。不同步骤有本质不同的质量要求（例如：摘要步骤要求"不抄原文"，实体对齐步骤要求"相关≠同一"），需要为每个步骤定义可操作的评估维度和判定规则。裁判模型（如Claude）逐条检查输出样本，给出结构化评估报告。

#### （2）树状搜索与动态变异策略

设计Prompt空间的树状搜索算法，核心机制包括：
- **动态变异**：根据裁判的失败分析，针对性地生成1-3个修复变体（而非随机扰动）
- **剪枝与回溯**：当子节点性能劣于父节点时标记为dead end，回溯到父节点尝试不同方向
- **收敛检测**：当所有评估维度通过时标记为converged，停止搜索

#### （3）非对称模型架构

提出**执行-裁判分离**的架构设计：
- **执行层**：轻量模型（GLM-4-Flash）执行Prompt，产生输出
- **裁判层**：强模型（Claude）评估输出质量，分析失败原因，指导下一轮变异

这种非对称设计解决了"弱模型无法自评"的问题，同时控制了计算成本——昂贵的强模型仅用于评估环节。

#### （4）Pipeline步骤间解耦优化

分析多步骤Pipeline中步骤间的耦合关系，论证在何种条件下各步骤可以独立优化，以及在何种条件下需要联合优化。本研究聚焦于**独立优化**场景，即每个步骤的Prompt优化可并行执行。

## 三、科学问题

### 核心问题

**在多步骤NLP Pipeline中，LLM裁判引导的树状搜索能否系统性地发现比人工设计更优的Prompt？**

### 具体科学问题

**SQ1：LLM裁判的评估有效性**
> 强模型（如Claude）作为裁判，其评估结果与人工评估的一致性如何？在格式正确性、语义完整性、边界情况处理三个维度上，LLM裁判是否可靠？

**SQ2：树状搜索的收敛性**
> 树状搜索在不同步骤、不同初始条件下是否总能收敛？收敛所需深度（迭代轮数）的分布如何？是否存在搜索空间中的局部最优陷阱？

**SQ3：动态变异 vs 随机搜索**
> 基于失败分析的动态变异策略是否显著优于随机扰动基线？变异方向的"解释性"（裁判给出的失败原因）对变异质量有何影响？

**SQ4：步骤专属评估标准的必要性**
> 使用步骤专属评估标准与使用通用评估标准（如BLEU、ROUGE）相比，最终Prompt的质量差异如何？通用指标能否替代步骤专属标准？

**SQ5：非对称架构的成本效益**
> 执行-裁判分离架构相对于"全流程使用强模型"方案，在最终输出质量和总计算成本之间的trade-off如何？

## 四、研究方案

### 4.1 实验设计

#### 4.1.1 目标Pipeline

以知识图谱构建Pipeline为研究对象，包含7个步骤：

| 步骤 | 功能 | 输入 | 输出 | 核心质量要求 |
|------|------|------|------|-------------|
| S1 | 记忆缓存更新 | 上轮缓存 + 文本窗口 | Markdown摘要 + 自我思考 | 不抄原文，保留结构标记 |
| S2 | 概念实体抽取 | 缓存 + 文本 | JSON数组 [{name, content}] | 高召回，消歧括号，描述充分 |
| S3 | 概念关系抽取 | 缓存 + 文本 + 实体列表 | JSON数组 [{e1, e2, content}] | 端点名称一致，内容具体 |
| S4 | 补充缺失实体 | 缓存 + 文本 + 名称列表 | JSON数组 [{name, content}] | 名称逐字匹配 |
| S5 | 实体内容增强 | 缓存 + 文本 + 实体信息 | JSON对象 {content} | 4个section齐全，不编造 |
| S6 | 实体对齐 | 当前实体 + 候选实体 | 裁决结果 | 合并判断准确 |
| S7 | 关系对齐 | 当前关系 + 已有关系 | 裁决结果 | 同性质才匹配 |

该Pipeline覆盖了NLP中典型的**生成**（S1）、**抽取**（S2-S4）、**增强**（S5）和**判别**（S6-S7）四类任务，具有代表性。

#### 4.1.2 数据

- **训练/搜索数据**：4部中文小说（红楼梦、三国演义、水浒传、西游记）的蒸馏数据，每个步骤20个测试窗口，分层采样
- **验证数据**：30个独立样本（与搜索数据不重叠），用于最终Prompt的泛化性验证
- **测试数据**：跨领域数据（技术文档、聊天记录），用于评估跨域泛化能力

#### 4.1.3 模型配置

| 角色 | 模型 | 用途 |
|------|------|------|
| 执行模型 | GLM-4-Flash | 运行Prompt变体，产生输出 |
| 裁判模型 | Claude (Sonnet) | 评估输出质量，分析失败，生成变体 |

### 4.2 对比方法

| 方法 | 描述 | 搜索策略 |
|------|------|---------|
| **TreePrompt**（本文） | LLM裁判 + 树状搜索 + 动态变异 | 树状DFS + 剪枝 |
| Flat-A/B | 传统A/B测试，预定义5个变体 | 平面枚举 |
| Random-Tree | 树状搜索 + 随机变异（无失败分析） | 树状DFS + 随机扰动 |
| DSPy-style | 基于程序化指标的自动优化 | 平面搜索 + 启发式指标 |
| Human-Crafted | 人工调优的Prompt（当前生产基线） | — |

### 4.3 评估指标

#### 定量指标

| 维度 | 指标 | 说明 |
|------|------|------|
| 格式正确性 | Parse Success Rate | 输出可正确解析为JSON的比例 |
| 抽取完整性 | Extraction Ratio | 抽取数量 / 参考数量（针对S2-S4） |
| 判别准确性 | Action Accuracy | 正确裁决比例（针对S6-S7） |
| 内容质量 | Weak Content Ratio | 空洞/模糊内容占比 |
| 名称一致性 | Name Match Rate | 端点名称与实体列表匹配率 |
| 搜索效率 | Nodes Explored | 收敛前探索的树节点总数 |
| 搜索深度 | Convergence Depth | 收敛时树的深度 |

#### 定性分析

- LLM裁判与人工评估在50个随机样本上的一致率（Cohen's Kappa）
- 树搜索路径的变异方向分析：哪些失败类型最常触发变异？哪些变异方向最常成功？
- 收敛Prompt相比初始Prompt的关键差异特征

### 4.4 实验流程

```
对每个步骤 S1-S7:
    1. 以当前生产Prompt作为树根节点 (depth=0)
    2. 执行层：用GLM-4-Flash在20个测试窗口上运行Prompt
    3. 裁判层：Claude逐条评估，输出结构化报告：
       - 格式通过率
       - 质量问题清单（附具体样本）
       - 失败原因分析
       - 变异方向建议
    4. 若所有维度通过 → 标记converged，返回该Prompt
    5. 若存在问题：
       a. Claude根据失败分析生成1-3个修复变体
       b. 对每个变体重复步骤2-3
       c. 若子节点优于父节点 → 继续深入
       d. 若子节点劣于父节点 → 标记dead end，回溯
    6. 终止条件（满足任一）：
       - 找到converged节点
       - 达到最大深度6
       - 所有路径均为dead end → 取全局最优
```

### 4.5 分析实验

#### 4.5.1 消融实验

| 实验 | 目的 |
|------|------|
| 去除裁判的失败分析（仅评分不分析） | 验证解释性反馈的价值 |
| 去除回溯机制（贪心搜索） | 验证回溯对避免局部最优的贡献 |
| 去除步骤专属标准（统一用通用指标） | 验证步骤专属评估的必要性 |
| 裁判模型降级（用同等级模型当裁判） | 验证非对称架构的必要性 |

#### 4.5.2 搜索行为分析

- 树结构可视化：每个步骤的搜索树形态（宽度、深度、dead end比例）
- 变异类型统计：增补规则 / 删减规则 / 重写规则 / 添加示例 各占比例
- 收敛速度：各步骤达到收敛所需的平均节点数

#### 4.5.3 泛化性验证

- 在验证集（30样本）上评估收敛Prompt的表现下降程度
- 在跨域数据（技术文档、聊天）上评估Prompt的迁移能力

### 4.6 预期贡献

1. **方法贡献**：提出LLM裁判引导的树状搜索Prompt优化框架，支持动态变异、自动回溯和收敛检测
2. **实证贡献**：在7步骤知识图谱构建Pipeline上验证框架有效性，提供每个步骤的收敛Prompt和质量对比
3. **分析贡献**：揭示Prompt空间中的搜索行为特征——变异方向的有效性分布、树搜索的收敛条件、步骤间优化难度差异

## 附录：初步实验结果

以下为基于知识图谱构建Pipeline的初步实验数据，用于说明方法可行性：

### 各步骤优化结果

| 步骤 | 关键改进 | 核心指标变化 |
|------|---------|-------------|
| S1 记忆缓存 | 添加章节标记规则，删除"锚点"触发词 | 问题数 40→5 (-87.5%) |
| S2 实体抽取 | 消歧括号取代类型标签，30字描述下限 | 内容质量显著提升 |
| S3 关系抽取 | "宁可多抽不可遗漏"+ content质量门控 | 弱内容 -71%，名称不匹配 -41% |
| S4 补充实体 | 强化content描述要求 | 描述性条目 9%→37% (+316%) |
| S5 内容增强 | 反幻觉原则，4-section结构，具体示例 | 编造内容大幅减少 |
| S6 实体对齐 | 向量搜索偏差警告，严格3条件关系规则 | 准确率 0.40→0.57 (+42.5%) |
| S7 关系对齐 | 核心谓语匹配规则，正反示例 | 准确率 0.75→0.87 (+16.0%) |

### 搜索效率

各步骤平均探索3-6个节点即收敛，最大搜索深度为5层。树状搜索相比平面枚举（需预定义并测试所有变体）在探索效率上有显著优势——仅沿着裁判判断为"有希望"的方向展开搜索。
