# Deep-Dream 开发蓝图

## 执行摘要

Deep-Dream 是面向 AI Agent 的“版本化记忆层”：把自然语言输入转化为可演化的知识图谱，并以“语义检索 + 图遍历 + 时间旅行 + 溯源解释”的组合能力，支持 Agent 完成 Remember（记住）与 Find（回忆），并通过 Dream（离线整合）持续发现隐含关联，使知识在长期运行中更可检索、更可解释、更可治理。本文档是高层概念设计蓝图（非实现方案），重点定义概念模型、架构边界、关键假设、治理与审计要求，以及面向开发者/产品/agent 的任务分解；所有基于未确认前提的建议均以“【基于假设】”标注。

关键结论要点  
- “Concept 单原语（万物皆概念）+ role（角色而非类型）”能显著压缩系统抽象面，但会把复杂度集中到“身份判定/合并治理/查询语义契约/访问控制”层；这些必须在 MVP 之前定稿，否则后续会在接口与数据一致性上付出高昂返工成本。  
- “版本（内容是否变化）与关联（是否被提及）解耦、关联无条件建立”是可审计性的核心，但会天然带来边数量增长与时间切片查询复杂化，需要明确“边的时间语义、分层视图与压缩策略”，否则可扩展性与成本不可控。【基于假设：输入规模可能持续增长】  
- “处理时间驱动的线性版本时间线”提供确定性与可回放性，但对“迟到事实/回填纠错/按真实发生时间检索”的表达力有限；是否引入双时间（record/valid）必须由产品需求明确，否则 Find 的语义会在后期变得不可兼容。citeturn0search3  
- Dream 产物与人工输入“同等地位”会显著放大伪关联风险；必须在概念层引入“候选/假设层、置信度与审核/撤销护栏”，并确保每条推断关系可追溯来源与生成上下文，否则会损害系统“可信赖”目标。【基于假设：Dream 会自动运行且覆盖面较大】  
- 需要将溯源从叙述升级为形式化数据模型：至少做到“任何知识可追溯到 observation 与产生行为（活动）/主体（agent）”；可参考entity["organization","万维网联盟（W3C）","web standards body"]的 PROV 概念模型（entity / activity / agent），作为可互操作的审计框架。citeturn0search0turn0search4  

## 目标、范围与关键假设

核心目标  
Deep-Dream 的产品目标是为 agent 提供一个“可长期运行”的外部记忆层，满足四个一等诉求：可检索（语义相关性与图遍历结合）、可进化（版本化演进与修正）、可溯源（可解释证据链）、可治理（置信度、纠偏、隔离与审计）。其中“时间维度”与“溯源链”不是附加功能，而是系统差异化能力的底座。

范围边界  
本蓝图覆盖 Remember →（抽取/对齐/版本/关联/溯源）→ 存储与索引 → Find（语义检索 + 图遍历 + 时间旅行 + 溯源解释）→ Dream（后台整合并生成关系）→ CLI（面向 agent 的可组合接口）→ 可视化（调试/验证工具）。不覆盖具体技术栈、模型选择、数据库选型、部署形态与实现细节。

显式关键假设（来自蓝图文本的直接设定）  
- 系统唯一原语是 Concept：实体、关系、原始输入（observation）本质同类。  
- 溯源内建：概念版本指向源 observation；即使内容不变也建立“提及关联”。  
- 版本（内容是否变化）与关联（是否被提及）解耦，且“与源文本的关联建立无条件、版本创建有条件”。  
- 语义是一等公民：Embedding 用于语义检索与概念对齐。  
- 每个概念携带可变置信度，随证据增减调整。  
- 版本演进按处理时间严格线性排序；事件时间仅作为内容属性；支持时间旅行查询。  
- 系统通过 CLI 暴露给 agent；输出机器可解析、命令可组合；支持多图谱隔离；可视化仅用于调试。

未显式说明但必须确认的假设（若不确认，将导致架构分歧）  
- 访问控制与隔离边界：多图谱隔离是否等同于“租户隔离/项目隔离/agent 隔离”？是否允许跨图谱共享记忆？【基于假设：存在多 agent 或多用户】  
- 数据敏感度与合规区域：是否涉及个人信息/机密数据？是否需要满足删除权、最小化、保留期等要求？（合规区域未指定，需确认）  
- Find 的产品契约：查询的输入形态（纯文本/结构化）、输出结构（必须携带哪些字段）、排序与可解释性要求、时间切片语义。  
- 质量目标：抽取准确率、对齐误差容忍度、Dream 伪关联阈值、检索召回/精度等如何评估；是否需要基准集与回归评测。【基于假设：需要可持续迭代与自动化验收】  
- 规模与成本：输入量、概念/边增长、索引更新频率、保留期、离线 Dream 频率（时间线、预算未指定，需确认）。  
- 运行模式：在线/离线、单机/分布式、是否要求长期无人值守（蓝图倾向无人值守，但需用 SLO/告警策略具体化）。  

未指定项（按约束统一标注）  
时间线、预算、团队规模、技术栈、合规区域：均为“未指定，需确认”。

## 概念模型与设计原则

概念实体  
Concept 是系统的唯一“知识单元”。为了让 Concept 在“可审计性、可治理性、agent 可用性”上可落地，建议在概念层明确以下最小语义契约（不等同于具体字段实现）：  
- 逻辑身份（Logical Identity）：跨版本不变，用于回答“还是不是同一个概念”。  
- 版本快照（Version Snapshot）：随处理时间线性演进，用于回答“在某时刻它是什么样”。  
- role（角色）：指示当前解释视角（如 observation / relation / entity），但不是刚性类型；role 可多重、可演化，必须可追踪其变化历史。【基于假设：role 会影响 UI/检索/治理策略】  
- 溯源链（Provenance）：将概念内容与其产生来源连接起来，至少覆盖“源 observation”与“产生/更新行为的责任主体”。参考 PROV 的核心抽象：provenance 可描述 entities、activities 与 people/agents，以支持对数据质量与可信度的判断。citeturn0search0turn0search4  

时间语义  
蓝图选取“处理时间（processing time）主导的线性时间线”，并将事件时间（event time）视为内容属性。这带来强确定性：任何时刻的系统状态可按处理顺序回放与查询。但产品上有一个必须前置决策：是否需要双时间（记录时间/有效时间）来支持回填纠错与按真实发生时间的点时查询。相关工程经验可参照entity["people","马丁·福勒","software author"]对 bitemporal history 的讨论：事件若要支持双时间，事件本身也需要携带两条时间轨。citeturn0search3  

溯源与审计  
Deep-Dream 的“可信赖”建立在可解释的证据链之上：对 agent 而言，“知道什么”不如“为什么这么知道、什么时候知道、从哪里知道”重要。建议将溯源能力分为三层视图：  
- 证据视图：概念 ↔ observation 的提及与派生关系（版本与关联均可追溯）。  
- 过程视图：抽取/对齐/合并/修正/Dream 生成作为 activity（行为）被记录，以支持审计与回放。【基于假设：需要对模型错误与治理决策进行追责/解释】  
- 主体视图：行为由哪个 agent/用户/系统任务触发（agent 作为 provenance 中的“责任主体”）。citeturn0search0turn0search4  

语义检索与向量依赖（外部依赖需验证）  
蓝图将语义空间视为“联想性”的基础：Embedding 将概念语义压缩为向量，使“按意义相关”成为可计算的距离。要让此能力可治理，建议在概念层明确：embedding 的模型版本、重算策略、漂移评估与可复现性要求。【基于假设：embedding 模型会迭代更新】  
在工程生态上，向量相似检索通常依赖 ANN 索引；例如 HNSW 家族以分层近邻图结构实现高效近似检索。citeturn1search2 相关工具如 Faiss 提供大规模向量相似检索与评估能力，可作为外部依赖候选（需进一步验证与选型）。citeturn1search1  

## 架构视图与模块划分

总体架构分层  
从 agent 可用性与治理视角，建议把 Deep-Dream 拆为五个概念层：  
- 接入层：CLI/接口契约（可组合、机器可解析、错误可编排）。  
- 注入层（Remember）：observation 纳管 → 概念抽取 → 身份对齐 → 版本/关联写入 → 溯源与置信度更新。  
- 存储与索引层：概念图、版本线、溯源图、语义索引（embedding）。  
- 召回层（Find）：语义检索与图遍历组合，并将时间切片与溯源解释作为一等输出。  
- 演化层（Dream）：离线发现潜在关系/孤立概念归属/跨域连接，并通过“候选层治理”进入主图。

## 依赖关系与潜在冲突

下表列出概念层需要提前对齐的“结构性张力点”，每项包含原因、影响范围、严重性与缓解建议；所有缓解均为概念级（不涉及实现细节）。

| 冲突/依赖点 | 原因 | 影响范围 | 严重性 | 建议缓解措施 |
|---|---|---|---|---|
| Concept 单原语 vs 可治理的分类/约束 | role 不是刚性类型会简化模型，但查询、授权、质量规则往往依赖稳定分类 | Find、权限、索引、可视化、治理 | 高 | 定义“最小 role 语义契约”：哪些 role 影响默认展示/召回/权限/保留策略；role 演化如何审计与回放。【基于假设：需要跨团队协作与长期维护】 |
| 版本严线性 vs 合并“类似 Git 分支”叙事 | 蓝图类比 Git 合并，但同时强调处理时间线性；历史合并语义若不明确会导致时间旅行解释矛盾 | 版本语义、溯源解释、审计 | 中 | 明确合并属于“身份合并（identity merge）”而非“历史分支合并（history merge）”；给出合并后时间线与溯源解释规则。【基于假设：会发生大量对齐不确定与后续合并】 |
| 处理时间主导 vs 事件时间检索诉求 | 处理时间确定但不等同事实发生时间；用户/agent 可能关心“当时发生了什么” | Find 语义、时间旅行、产品预期 | 高 | 通过 ADR 决定是否引入双时间；若不引入，需在产品层明确限制与用户心智，并在输出中显式区分 processing_time 与 event_time。citeturn0search3 |
| 关联无条件建立 vs 边数量膨胀与快照复杂化 | 每次提及都建关联会导致边线性增长；时间切片需要定义边的可见性与压缩 | 成本、性能、可扩展性 | 高 | 引入“边分层视图”：事实边/提及边/候选边分层治理；对提及边定义窗口化汇总、抽样或可折叠视图。【基于假设：输入规模持续增长】 |
| Dream=同等地位 vs 可信边界 | Dream 可能产生伪关联；若与事实同权会污染记忆与推理 | 可信性、治理、安全 | 高 | 默认将 Dream 写入“候选/假设层”，并要求：溯源、置信度、可撤销与复核流程；默认 Find 召回对候选层降权或需显式请求。【基于假设：Dream 自动或频繁运行】 |
| 历史不可删（淡出） vs 隐私/合规删除权 | “历史不可篡改”提升审计，但个人数据可能需要删除或最小化 | 合规、风险、运维 | 高 | 在概念层预留“可遗忘/可加密删除/可匿名化”的治理语义；若合规区域涉及 GDPR/UK GDPR，应能响应删除请求（删除权在相关法规文本中被明确为数据主体权利）。【基于假设：处理个人数据且适用相关法规】citeturn3search8turn3search14 |
| 多图谱共享愿景 vs 访问控制与越权召回 | 共享记忆空间会引入跨主体数据混用风险 | 安全、隐私、产品边界 | 高 | 在图谱命名空间之上定义“主体-权限-审计”三元：谁可写、谁可读、谁可读溯源原文；Find 默认遵循最小权限原则。【基于假设：多用户/多 agent 共用】 |
| 语义检索依赖外部模型 vs 可复现与漂移治理 | embedding 模型迭代会导致检索排序漂移，影响回归与审计 | 质量、回归测试、运维 | 中 | embedding 版本化与评估基准前置：记录模型版本与重算策略；关键查询具备回归集；必要时保留历史向量以支持解释。【基于假设：模型会更新】 |

## 未覆盖点与风险治理

以下风险按安全、隐私、合规、性能、可扩展性、运维、成本、数据质量维度整理；对策均为概念级，并在依赖外部条件时标注“【基于假设】”。

| 风险/缺口 | 维度 | 风险等级 | 概念性对策 |
|---|---|---|---|
| Find 的查询契约不完整（输入形态、输出结构、时间切片语义、解释字段） | agent 可用性 | 高 | 先定义“可组合查询代数”：语义检索→过滤/聚合→图遍历→时间切片→溯源解释→置信度展示；把时间与溯源作为一等输出结构（不可选）。 |
| 概念身份判定误差导致误合并/误拆分，并在溯源与检索中放大 | 数据质量 | 高 | 合并必须可解释、可逆或可隔离；默认保守（不确定则新建）；建立“合并候选队列 + 复核/回滚”治理闭环。【基于假设：对齐存在长期不确定性】 |
| Dream 伪关联污染“事实层”，影响 agent 决策 | 可信性/安全 | 高 | 设立候选层：Dream 默认写入候选关系；默认 Find 不把候选层当事实；候选层提升到事实层需要证据累积/复核。 |
| Prompt injection 与敏感数据泄露（observation 中包含指令样式内容，被召回后影响下游） | 安全 | 高 | 记忆内容必须视为“不可信数据”；召回时强制携带来源与置信度，不可直接作为系统指令执行；对高风险内容分类/隔离。可参考entity["organization","开放式网络应用安全项目（OWASP）","appsec nonprofit"]对 LLM 应用风险（如 Prompt Injection、敏感信息泄露等）的总结。citeturn1search8turn1search4 |

| 关联无条件建立导致边增长，索引与存储成本不可控 | 成本/可扩展性 | 高 | 分层存储与视图：提及边可折叠/汇总；为时间旅行定义默认粒度（例如按日/按批次）；对低价值关联进行压缩策略。【基于假设：数据规模增长快】 |
| 向量检索外部依赖的性能与准确率难以预测 | 性能 | 中 | 在产品层定义可接受的召回/延迟区间；构建评测集；把 ANN 索引能力视为可替换依赖（例如 HNSW 类方法在研究与工程中广泛使用）。citeturn1search2 |

| 多图谱隔离缺少授权与审计，可能出现越权召回 | 安全/隐私 | 高 | 增加“主体模型 + 权限决策 + 审计日志”；Find 的任何结果都必须通过权限过滤，并对“溯源原文”的展示施加更严格控制。【基于假设：多用户或多组织使用】 |

## 关键决策点与备选方案

本节聚焦“会改变系统语义或治理边界”的决策；每项给出 2–3 个备选与权衡，以便产品/开发/agent 共同对齐。

时间模型决策  
- 方案 A：仅处理时间线性版本（蓝图默认）  
  - 优点：确定性强、时间旅行易解释、审计回放简单。  
  - 代价：对按事件真实发生时间的点时查询与迟到纠错能力弱。  
  - 适用：【基于假设：用户主要关心“系统何时知道”而非“真实何时发生”】【基于假设：迟到数据较少】  
- 方案 B：双时间（record/valid）  
  - 优点：同时回答“何时记录/何时有效”，支持回填纠错与合规审计语义。  
  - 代价：模型与查询复杂度显著上升（事件本身也需双时间）。citeturn0search3  
  - 适用：【基于假设：事件时间查询是核心需求或回填频繁】  
- 方案 C：事件溯源视图（把变更作为事件流，快照为派生）  
  - 优点：审计回放天然；可派生多种视图。  
  - 代价：需要更强的一致性治理与派生解释能力。【基于假设：审计/回放优先级极高】  

图查询抽象与生态对齐  
- 方案 A：内部保持 Concept 图抽象，外部暴露稳定“查询代数”（不绑定具体图语言）  
  - 优点：接口稳定、可替换底层实现、适合 CLI 组合。  
  - 代价：需要维护映射层语义一致性。  
- 方案 B：属性图（property graph）优先，参考 GQL 标准语义  
  - 优点：与行业标准对齐；生态成熟；遍历与路径表达直观。citeturn0search1  
  - 代价：要明确“关系作为 Concept”时的建模规则（reification 与查询表达）。  
- 方案 C：RDF 语义网路线（triple graph）  
  - 优点：标准化强；与语义互操作生态匹配；RDF 抽象语法清晰。citeturn0search2  
  - 代价：关系一等化与版本/时间语义往往更复杂；学习成本上升。  

Dream 治理策略  
- 方案 A：Dream 产物直接进入事实层（同权）  
  - 优点：图谱增长快、联想更强。  
  - 代价：伪关联风险高，损害可信与可治理性（不建议默认）。  
- 方案 B：Dream 写入候选层，需证据/复核提升为事实  
  - 优点：兼顾创造性与可信性；可撤销；利于评估。  
  - 代价：需要候选层管理与提升规则。【基于假设：系统追求“可信赖”优先】  
- 方案 C：Dream 仅按需触发（Agent 手动或在特定条件下）  
  - 优点：风险与成本可控，适合 MVP。  
  - 代价：演化速度慢；跨域连接发现能力弱。【基于假设：早期资源有限】  

溯源模型形式化程度  
- 方案 A：最小溯源（版本/提及关联 → observation）  
  - 优点：简单，足以支撑“从概念回到原文”。  
  - 代价：难以表达“活动/主体/派生链”的审计需求。  
- 方案 B：PROV 风格（entity/activity/agent）  
  - 优点：可互操作、可审计、利于质量评估；W3C 提供成熟概念框架。citeturn0search0turn0search4  
  - 代价：元信息更多，治理更重。【基于假设：面向企业/合规/可追责场景】
